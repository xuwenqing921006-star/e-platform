# Bug 修复报告：后台登录 5176 端口 CORS

## 用户原话

> 登录后台返回Invalid CORS request

## 经验回查

- 是否已有相关经验：是
- 相关经验：
  - `T-009: 若依安全、契约响应与 H5 公开路由适配`
  - `Bugfix: 后台登录、县域内容分类、多联系人和 H5 返回上下文`
  - `Bugfix: 后台业务账号路由权限与登录错误提示`
- 为什么仍然犯错：已有经验覆盖了登录契约、验证码禁用、错误提示和若依权限桥接，但没有把“前后端并行启动时后台临时端口变更也必须进入 CORS 白名单和启动文档”写成可执行验收项。

## 问题分析

- 现象：后台从 `5176` 端口登录时，后端返回 `Invalid CORS request`。
- 影响范围：本地同时启动 H5 `5175` 和若依后台 `5176` 的用户验收环境。
- 直接原因：`ResourcesConfig` 的 CORS 白名单只包含 `5175`、`5180`、`5199`，缺少 `5176`。
- 根本原因：启动文档中曾把 H5 和若依后台都写为用户验收 `5175`，实际并行启动时后台改用 `5176`，但 CORS 配置和回归测试没有同步覆盖该端口。

## 修复方案

- 修改文件：
  - `backend/ruoyi-framework/src/main/java/com/ruoyi/framework/config/ResourcesConfig.java`
  - `backend/ruoyi-admin/src/test/java/com/ruoyi/web/security/SecurityContractTest.java`
  - `docs/startup.md`
- 修改说明：
  - 为 `http://localhost:5176` 和 `http://127.0.0.1:5176` 增加后端 CORS 放行。
  - 增加安全契约测试，防止 5176 验收端口再次漏配。
  - 将启动文档中的若依后台用户验收端口同步为 `5176`。

## 验证方式

- 前端验证：
  - 打开 `http://127.0.0.1:5176/`，执行后台登录操作，不应再出现 `Invalid CORS request`。
- 终端验证：
  - CORS 预检：`OPTIONS http://127.0.0.1:8003/login` 携带 `Origin: http://127.0.0.1:5176` 返回 200，并包含对应 `Access-Control-Allow-Origin`。
  - 后台代理登录：`POST http://127.0.0.1:5176/dev-api/login` 不再返回 CORS 错误；错误账号密码返回业务错误响应。
  - Maven 回归：`backend/.mvnw.cmd -q -pl ruoyi-admin -am "-Dtest=SecurityContractTest#securityConfigDeclaresPublicAndProtectedApiRoutes" "-Dsurefire.failIfNoSpecifiedTests=false" test` 通过。
