# Bugfix: H5 与后台乱码、卡片越界和原型外入口残留

## 触发

用户反馈：

1. H5 页面“乡村振兴与金融服务”下的服务指引卡片长度超出边界，且卡片列表内容乱码。
2. 后台所有页面全是乱码。
3. 后台出现很多原型以外的内容、字段和模块，例如导航栏、头部搜索、布局大小、公告通知等。

## 结论

本次问题不是单一前端渲染问题，而是三类问题叠加：

- Docker MySQL 首次初始化 SQL 未显式设置客户端字符集，当前库中中文数据被按错误字符集写入，导致 H5 与后台接口返回乱码。
- H5 卡片缺少 `min-width: 0`、换行和溢出约束，长标题或异常字符串会撑破移动端边界。
- 若依后台仍保留了部分模板外壳和默认菜单能力，没有把可见入口收敛到本项目原型确认的业务范围。

## 修复

- 在所有 Docker 初始化相关 SQL 前增加 `SET NAMES utf8mb4;`，避免后续新库初始化时再次写入乱码。
- 新增 `central_bank_repair_mojibake_mysql.sql`，用于修复当前已写坏的 MySQL 数据。
- 新增 `central_bank_admin_menu_cleanup.sql`，默认隐藏若依模板菜单，仅保留内容管理、金融产品、账号管理、操作日志，Excel 导入保留为隐藏路由。
- 已对当前 Docker MySQL 执行乱码修复和菜单清理。
- 收敛 H5 内容卡片和产品卡片的移动端溢出样式。
- 清理若依后台头部原型外入口：搜索、全屏、布局大小、公告通知、锁屏、个人中心、布局设置。
- 将“修改密码”保留为用户下拉菜单唯一业务入口之一，退出登录保留。
- 将若依个人中心页面收敛为仅修改密码，删除头像、邮箱、手机号、部门、角色、创建时间等原型外字段。
- 扩展 `scripts/audit-mock-exit.mjs`，把乱码风险、后台默认菜单、头部默认工具和 H5 卡片溢出纳入审计。

## 验证

- `node scripts/audit-mock-exit.mjs` 通过。
- `npm.cmd run typecheck` 通过。
- `npm.cmd run build:prod` 通过，仅有若依前端常规包体积提示。
- 当前 MySQL 直接查询确认中文恢复正常。
- 当前公开产品接口返回中文正常，例如 `建设银行:善营贷`、`阳光惠农贷:阳光惠农贷`、`中信银行:粮农贷`。
- 当前后台可见菜单确认仅有内容管理、金融产品、账号管理、操作日志；Excel 导入为隐藏路由。

## 防复发

- 含中文的 MySQL 初始化脚本必须显式声明客户端字符集。
- Docker 首次初始化后，不能只看页面；必须直接查询数据库或接口响应确认中文没有 mojibake。
- 后台若使用若依基座，默认菜单、头部工具和个人资料页都必须被显式审计，不得只依赖“菜单没有显示”。
- H5 列表卡片必须具备移动端换行与溢出保护，尤其是标题、标签和时间字段。
